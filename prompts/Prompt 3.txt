Objetivo da fase

Implementar autenticação segura para usuários administrativos e estrutura base de RBAC, com login no painel admin, proteção de rotas e recuperação de senha.

Escopo obrigatório
Criar módulo auth no NestJS.
Criar módulo users.
Implementar login por e-mail e senha.
Implementar hash de senha com bcrypt ou argon2.
Implementar access token JWT de curta duração.
Implementar refresh token seguro.
Preferir refresh token em cookie:
HttpOnly;
Secure em produção;
SameSite;
expiração configurável.
Implementar JwtAuthGuard.
Implementar RolesGuard ou PermissionsGuard.
Criar decorators:
@CurrentUser;
@RequirePermissions;
@Public, se necessário.
Implementar endpoints:
POST /api/v1/auth/login;
POST /api/v1/auth/refresh;
POST /api/v1/auth/logout;
POST /api/v1/auth/forgot-password;
POST /api/v1/auth/reset-password;
GET /api/v1/auth/me.
Criar tela de login em apps/admin.
Criar proteção básica no layout administrativo.
Registrar eventos importantes em audit_logs.
Consultar permissões e papéis a partir do MySQL via Prisma.
Preparar estrutura de autenticação reaproveitável para área do doador em fases futuras, sem implementar toda a área do doador agora.
Correções obrigatórias
Mensagens de login não devem revelar se o e-mail existe.
Não logar senha, token, refresh token ou dados sensíveis.
Preparar estrutura para 2FA futuro, mas não implementar 2FA nesta fase.
A área do doador ainda não precisa ter login completo nesta fase, salvo estrutura mínima compartilhável.
Não criar nenhum fluxo financeiro nesta fase.
Não alterar a modelagem de dados da Fase 2 sem justificar e pedir confirmação.
O que não implementar agora

Não implementar doações, Asaas, Pix, dashboard financeiro, dashboard do doador, carteirinha, animais, campanhas, portal de transparência ou PWA.

Segurança obrigatória
Rate limit básico no login.
Proteção contra enumeração de usuários.
Tokens com expiração configurável por .env.
Secrets vindos exclusivamente de .env.
Guardas aplicados em rotas administrativas.
CORS compatível com apps/admin e apps/web.
Senhas nunca devem ser retornadas em DTOs.
Hash de refresh token deve ser armazenado de forma segura, se persistido.
Logout deve invalidar o refresh token quando aplicável.
Reset de senha deve usar token temporário com expiração e uso único.

Não criar novas tabelas sem necessidade. Se for indispensável criar tabela para refresh tokens ou password reset tokens, justifique tecnicamente e crie migration compatível com MySQL.

Endpoints esperados
POST /api/v1/auth/login

Entrada:

email;
password.

Saída:

dados básicos do usuário;
permissões;
access token;
refresh token via cookie seguro ou estratégia definida.
POST /api/v1/auth/refresh

Renova access token usando refresh token válido.

POST /api/v1/auth/logout

Invalida a sessão atual.

GET /api/v1/auth/me

Retorna usuário autenticado, papéis e permissões.

POST /api/v1/auth/forgot-password

Gera token de recuperação e prepara envio por e-mail.

POST /api/v1/auth/reset-password

Valida token de recuperação e altera senha.
