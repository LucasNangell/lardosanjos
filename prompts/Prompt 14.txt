## Objetivo da fase

Criar o módulo administrativo para cadastrar e gerenciar os dados Pix usados na geração de doações únicas por BR Code/EMV sem Asaas.

## Escopo obrigatório

1. Criar página admin:

   * `/admin/configuracoes/pix`;
   * ou equivalente dentro de `apps/admin`.
2. Criar endpoints:

   * `GET /api/v1/admin/pix/settings`;
   * `PUT /api/v1/admin/pix/settings`;
   * `POST /api/v1/admin/pix/settings/test`.
3. Campos obrigatórios:

   * nome do recebedor;
   * cidade do recebedor;
   * chave Pix;
   * tipo da chave;
   * descrição padrão;
   * txid padrão, se aplicável;
   * valor mínimo;
   * valores rápidos;
   * permitir valor personalizado;
   * instruções públicas;
   * exigir dados do doador;
   * exigir comprovante;
   * ocultar dados sensíveis;
   * status ativo/inativo;
   * ambiente.
4. Criar botão “Gerar QR Code de teste”.
5. Exibir Pix Copia e Cola de teste.
6. Validar campos antes de salvar.
7. Registrar toda alteração em `audit_logs`.

## Correções obrigatórias

1. Apenas usuário com `PIX_SETTINGS_WRITE` pode alterar.
2. Usuário com permissão insuficiente não pode ver chave completa se a regra exigir ocultação.
3. Alterar Pix deve atualizar imediatamente o checkout público.
4. Não implementar confirmação de doações nesta fase.
5. Não chamar Asaas.
6. Não armazenar chave Pix em local diferente de `pix_settings`.

## O que não implementar agora

Não implementar tela de confirmação Pix, transparência, mural ou relatórios.

## Segurança obrigatória

1. Mascarar chave Pix quando apropriado.
2. Auditar leitura/alteração de dados sensíveis.
3. Validar tipo de chave Pix.
4. Não logar chave completa desnecessariamente.
5. Proteger rota por RBAC.
6. Não permitir configuração ativa inválida.
7. Não expor chave completa em endpoint público se `hide_sensitive_details` estiver ativo.

## Testes obrigatórios

1. Admin autorizado salva configuração.
2. Usuário sem permissão é bloqueado.
3. Chave inválida é rejeitada.
4. QR Code de teste é gerado.
5. Alteração aparece em auditoria.
6. Checkout público usa nova configuração.
7. Build e testes passam.
8. Dados são persistidos no MySQL.
9. Não há chamada ao Asaas.

## Critérios de aceite

1. Administração consegue configurar Pix.
2. Sistema gera teste BR Code válido.
3. Permissões funcionam.
4. Auditoria registra alterações.
5. Checkout público respeita nova configuração.
