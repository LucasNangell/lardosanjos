## Objetivo da fase

Implementar o checkout de doação mensal recorrente, criando assinatura no Asaas e registro local em `subscriptions`.

## Escopo obrigatório

1. Criar página pública:

   * `/seja-um-anjo`;
   * checkout de assinatura em etapas.
2. Listar planos ativos de `donation_plans`.
3. Permitir valor personalizado, se previsto.
4. Coletar dados do doador:

   * nome;
   * e-mail;
   * WhatsApp;
   * CPF/CNPJ, se necessário;
   * endereço, se necessário;
   * consentimentos LGPD;
   * preferência de mural;
   * anonimato.
5. Criar endpoint:

   * `POST /api/v1/public/donations/subscription`.
6. Criar ou atualizar donor local no MySQL.
7. Criar ou atualizar customer no Asaas.
8. Criar assinatura mensal no Asaas.
9. Persistir em:

   * `donors`;
   * `subscriptions`;
   * possivelmente `payments` inicial, se a API retornar cobrança inicial.
10. Exibir tela de confirmação com status pendente até webhook confirmar.
11. Criar estados de falha e tentativa novamente.

## Correções obrigatórias

1. Não tratar assinatura criada como pagamento confirmado.
2. A ativação plena deve depender de webhook ou consulta segura.
3. Não usar Pix avulso manual para assinatura.
4. Se Pix recorrente/Pix Automático estiver indisponível, tratar como recurso opcional/futuro.
5. Não armazenar cartão no banco local.
6. Não alterar o fluxo de Pix avulso único.
7. Não criar assinatura sem consentimento obrigatório aceito.

## O que não implementar agora

Não implementar área do doador completa, upgrade, cancelamento, webhooks, carteirinha, transparência ou mural.

## Segurança obrigatória

1. Não expor Asaas no frontend.
2. Não logar cartão.
3. Validar todos os dados no backend.
4. Registrar consentimentos importantes.
5. Política de privacidade deve ser referenciada no checkout.
6. Não salvar CVV, número completo do cartão ou dados sensíveis desnecessários.

## Testes obrigatórios

1. Criar assinatura com plano fixo.
2. Criar assinatura com valor personalizado, se habilitado.
3. Simular cartão recusado.
4. Simular falha Asaas.
5. Validar consentimentos obrigatórios.
6. Garantir que assinatura pendente não aparece como ativa.
7. Garantir que Pix avulso não é usado.
8. Validar gravação em `donors` e `subscriptions` no MySQL.
9. Validar que `asaas_subscription_id` é armazenado corretamente.

## Critérios de aceite

1. Usuário consegue iniciar assinatura mensal.
2. Asaas retorna ID de assinatura em sandbox.
3. Banco local armazena `asaas_subscription_id`.
4. Status inicial é coerente.
5. Build e testes passam.
6. Nenhum pagamento pendente é tratado como confirmado.
