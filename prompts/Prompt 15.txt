## Objetivo da fase

Criar o painel administrativo para confirmar, rejeitar, marcar como duplicada e auditar doações Pix avulsas feitas sem Asaas.

## Escopo obrigatório

1. Criar página admin:

   * `/admin/financeiro/pix-confirm`;
   * ou equivalente.
2. Criar endpoints:

   * `GET /api/v1/admin/pix/donations`;
   * `GET /api/v1/admin/pix/donations/:id`;
   * `POST /api/v1/admin/pix/donations/:id/confirm`;
   * `POST /api/v1/admin/pix/donations/:id/reject`;
   * `POST /api/v1/admin/pix/donations/:id/mark-duplicate`;
   * `POST /api/v1/admin/pix/donations/:id/request-info`.
3. Criar tabela com:

   * data;
   * nome;
   * e-mail;
   * WhatsApp;
   * valor;
   * status;
   * comprovante;
   * mensagem;
   * data em que marcou “Já fiz o Pix”;
   * admin responsável;
   * ações.
4. Criar filtros:

   * status;
   * período;
   * valor;
   * com comprovante;
   * sem comprovante;
   * nome/e-mail/telefone.
5. Criar visualizador seguro de comprovante.
6. Confirmar pagamento:

   * mudar para `CONFIRMADO_MANUALMENTE`;
   * registrar admin;
   * registrar data;
   * registrar histórico em `pix_payment_confirmations`;
   * atualizar totais financeiros confirmados;
   * liberar para transparência;
   * liberar para mural, se consentido.
7. Rejeitar pagamento:

   * exigir motivo;
   * registrar auditoria;
   * não contabilizar.
8. Marcar duplicado:

   * registrar auditoria;
   * não contabilizar.
9. Solicitar informações:

   * preparar envio de e-mail, se serviço já existir;
   * caso contrário, registrar pendência clara.

## Correções obrigatórias

1. Apenas `PIX_CONFIRM_MANUAL` pode confirmar.
2. Comprovantes não podem ser públicos.
3. Pix pendente não entra na transparência.
4. Rejeitado não entra no mural.
5. Não sobrescrever status confirmado sem auditoria.
6. Toda transição deve ser auditada.
7. Não chamar Asaas.
8. Usar transações no MySQL/Prisma quando a ação alterar múltiplas tabelas.

## O que não implementar agora

Não implementar integração automática com extrato bancário, Open Finance, OFX ou CNAB.

## Segurança obrigatória

1. RBAC rigoroso.
2. URL assinada temporária para comprovante.
3. Auditoria de confirmação/rejeição.
4. Proteção contra confirmação duplicada.
5. Transações no banco para evitar inconsistência.
6. Não expor comprovantes a usuários públicos.
7. Não permitir confirmação de Pix já rejeitado/duplicado sem fluxo explícito de revisão.

## Testes obrigatórios

1. Listar Pix pendentes.
2. Confirmar Pix.
3. Rejeitar Pix.
4. Marcar duplicado.
5. Usuário sem permissão não confirma.
6. Confirmado aparece como receita confirmada.
7. Pendente não aparece como receita confirmada.
8. Rejeitado não aparece no mural.
9. Comprovante só é acessível por admin autorizado.
10. Alterações são gravadas em `pix_payment_confirmations`.
11. Dados são persistidos corretamente no MySQL.

## Critérios de aceite

1. Admin confirma/rejeita Pix corretamente.
2. Transparência só recebe Pix confirmado.
3. Auditoria registra tudo.
4. Build e testes passam.
5. Não há dependência do Asaas no Pix manual.
